Overview

In PA1, you extended xv6 by implementing new system calls and maintaining per-process system call accounting. In PA2, you redesigned the xv6 scheduler and implemented a System-Call-Aware Multi- Level Feedback Queue (SC-MLFQ) scheduler.

In this assignment, you will extend xv6's virtual memory subsystem by implementing page replacement when physical memory becomes full. The xv6 repository used in this course already implements lazy allocation: when a process accesses a previously unmapped virtual page, a page fault occurs and the kernel allocates a physical page dynamically. However, if the system runs out of physical memory, xv6 currently kills the process.

Your task in this assignment is to modify xv6 so that instead of terminating the process, the kernel evicts an existing page using a page replacement algorithm, stores its contents in a swap area, and reuses the freed frame. When a swapped-out page is accessed again, it must be restored from swap.

This assignment builds directly on concepts discussed in class, including:

* Virtual memory and paging
* Page faults and trap handling
* Page replacement policies such as LRU and CLOCK

Your implementation must also integrate components developed in earlier assignments:

* Process statistics from PA1
* Scheduling priorities from PA2

Assignment Goal

Extend xv6 so that:

1. Physical memory is limited to a fixed number of frames.
2. When all frames are occupied, the kernel selects a victim page using the Clock page replacement algorithm.
3. The victim page is evicted and its frame reused.
4. Eviction decisions take into account process priority from the SC-MLFQ scheduler implemented in PA2.
5. The kernel maintains per-process memory statistics similar to the accounting introduced in PA1.



Memory System Behaviour

In the current xv6 repository used for this course:

* sbrk() increases the process address space without allocating pages immediately.
* When the process first accesses a page, a page fault occurs.
* The kernel allocates a physical page in the page fault handler.

However, if physical memory is exhausted, kalloc() fails and the process is terminated. In this assignment, you will modify this behaviour so that when memory is full:
* an existing page is evicted
* its physical frame is reused for the new page
* the contents of the evicted page are stored in a swap area
* if the evicted page is accessed again, it is restored from swap

Frame Table

To manage page replacement, you must maintain a global frame table describing all physical frames allocated to user processes.

Each frame entry should store metadata such as:

* whether the frame is currently in use
* which process owns the page
* the virtual address mapped to the frame
* a reference bit used by the Clock algorithm

This frame table will allow the kernel to efficiently determine which page occupies each frame.



Page Replacement Policy

You must implement the Clock page replacement algorithm. The algorithm operates as follows:
1. Maintain a circular list of frames.
2. Each frame has a reference bit.
3. When searching for a victim frame:
o If the reference bit is 0, the page is selected for eviction.
o If the reference bit is 1, the bit is cleared and the algorithm continues scanning.

When a victim page is selected:

* the page contents must be copied to swap space
* the page table entry must be updated
* the frame must be released
* the frame may then be reused for the new page



Swap Space

In addition to page replacement, you must implement a simple swap mechanism.

* When a page is evicted from memory, its contents must be stored in a swap space.
* When the page is accessed again, it must be restored from swap.
* Swap space should be implemented as a fixed-size array in memory.
* You are not required to use the file system or disk for this assignment.

Scheduler-Aware Replacement (Integration with PA2)

Your replacement algorithm must consider the SC-MLFQ scheduling level of processes. If multiple candidate pages exist for eviction:
* pages belonging to lower priority processes (lower MLFQ queues) should be preferred for eviction.

This ensures that interactive processes retain their working sets longer.

These statistics must be updated during page fault handling and page replacement.



New System Call

1. int getvmstats(int pid, struct vmstats *info) Description
Retrieves virtual memory statistics of the process with the given PID. The kernel must populate the following structure:
struct vmstats {
int page_faults; int pages_evicted;
int pages_swapped_in; int pages_swapped_out; int resident_pages;
};

Return Value

• 0 on success
• -1 if PID invalid

This system call behaves similarly to getmlfqinfo() implemented in PA2.
